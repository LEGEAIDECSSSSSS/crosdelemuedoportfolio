Applying Filters to SQL Queries

Data security and access control are critical aspects of any organization's IT infrastructure. In this project, I utilized SQL queries to filter and analyze login attempts and employee data to enhance security monitoring and facilitate system updates. By applying filters with SQL operators such as AND, OR, NOT, and LIKE, I was able to extract meaningful insights, detect suspicious activities, and streamline administrative tasks. The following sections detail the various filtering techniques used and their practical applications in real-world scenarios.

Summary

  1. Failed Login Attempts After Hours

    2. Problem Statement: A potential security breach may have occurred outside of regular working hours (after 18:00). To investigate unauthorized access attempts, I needed to filter out all failed login attempts that took place after business hours.

    • Used WHERE clause with AND to find failed logins after 18:00.
    • Query filters unsuccessful attempts for security review.

      • Read Full Query →

      SELECT * FROM log_in_attempts 
WHERE login_time > '18:00' AND success = FALSE;

Explanation: 
- This query filters login attempts that:
- Happened after 18:00.
- Were unsuccessful.
- This helps identify unauthorized access attempts outside working hours.
  2. Login Attempts on Specific Dates

    3. Problem Statement: A suspicious event was reported on May 9, 2022. To analyze any potential security threats, I needed to retrieve all login attempts that occurred on May 9, 2022, and the day before (May 8, 2022), to check for any unusual activity leading up to the event.

    • Investigated suspicious logins on May 8–9, 2022.
    • Used WHERE clause with OR to extract logs.

      • Read Full Query →

      SELECT * FROM log_in_attempts 
WHERE login_date = '2022-05-09' OR login_date = '2022-05-08'; 

Explanation:
- This query extracts all login attempts that occurred on either of the two 
  specified dates using the OR operator. 
- This helps track any unusual login patterns leading up to the event.
  3. Login Attempts Outside Mexico

    4. Problem Statement: Unauthorized logins from foreign locations posed a security risk to the organization. I needed to extract all login attempts that occurred outside of Mexico to identify potential account compromise or unauthorized access from international locations.

    • Flagged all non-Mexico logins for investigation.
    • Used WHERE clause with NOT LIKE to extract logs.

      • Read Full Query →

      SELECT * FROM log_in_attempts 
WHERE country NOT LIKE 'MEX%'; 

Explanation: 
- This query selects all login attempts excluding those from Mexico. 
- The NOT LIKE 'MEX%' condition ensures that logins from "MEX" and "MEXICO" are 
  filtered out, leaving only logins from other countries for further investigation.
  4. Filtering Employees by Department

    5. Problem Statement: The IT team needed to update computers for employees in Marketing, Finance, Sales, and IT. Different security updates were required based on department.

    • Retrieved lists of employees in Marketing, Finance, Sales, and IT for system updates.
    • Used WHERE, LIKE, AND, and OR operators.

      • Read Full Query →

      SELECT * FROM employees 
WHERE department IN ('Marketing', 'Finance', 'Sales', 'IT');

Explanation:
- Uses the IN operator to retrieve employees from multiple departments at once.
- Instead of multiple OR conditions, IN makes the query more readable and efficient.
- Ensures that only employees from these four departments are included for security updates.

Conclusion

In this project, I demonstrated how to apply SQL filtering techniques to extract security-critical data efficiently. By utilizing SQL operators like AND, OR, NOT, and LIKE, I was able to identify suspicious login attempts, analyze login trends, and retrieve employee data based on departmental needs. These queries play a crucial role in strengthening security monitoring, helping organizations proactively respond to potential threats and maintain data integrity.

File Permissions in Linux

Managing file permissions is essential for securing sensitive data. In this project, I used Linux commands to check and modify file permissions, ensuring proper authorization. Below are the key steps I took.

Summary

  1. Checking File and Directory Permissions

    2. Problem Statement: I needed to verify permissions on important files in the projects directory.

    • Used `ls -la` to check file and directory permissions.

      • Read Full Command →

                      ls -la
                Explanation:
                - Lists all files and directories with their permissions.
                - Reveals hidden files by using the `-a` flag.
  2. Changing File Permissions

    3. Problem Statement: Some files allowed unwanted write access. I needed to modify their permissions.

    • Used `chmod` to remove write permissions from unauthorized users.

      • Read Full Command →

                      chmod o-w project_k.txt
                Explanation:
                - This command removes write (`w`) permission from "others".
                - Ensures that only the owner and group can modify the file.
  3. Restricting Access to a Directory

    4. Problem Statement: Only `researcher2` should access the `drafts` directory.

    • Used `chmod` to remove execute permissions from unauthorized users.

      • Read Full Command →

                      chmod g-x,o-x drafts/
                Explanation:
                - Removes execute permissions (`x`) for group and others.
                - Only the owner (`researcher2`) retains access.

Conclusion

This project highlights how file permissions in Linux can enhance security. By checking permissions, modifying access levels, and enforcing proper restrictions, I ensured sensitive data remained protected.

Incident Response Case Studies

Below are detailed case studies showcasing my incident handling and response processes in real-world scenarios.

Case Study 1: Ransomware Attack on a Healthcare Company

Overview: A healthcare organization experienced a ransomware attack that encrypted critical files, disrupting operations. The incident response team had to quickly assess the situation, contain the threat, and determine the best course of action.

Incident Breakdown:

  • Detection & Analysis: The attack was detected when multiple employees were locked out of systems, and ransom notes demanding payment appeared.
  • Containment & Eradication: Infected systems were shut down, and external cybersecurity experts were engaged to assist.
  • Recovery: With backups unavailable, the organization had to choose between paying the ransom or pursuing alternative recovery methods.

Lessons Learned & Preventive Measures:

  • Implement a robust backup strategy to prevent data loss.
  • Conduct regular phishing awareness training for all employees.
  • Enforce network segmentation to limit the spread of malware.
Case Study 2: Network Traffic Analysis with Wireshark

Overview: Network traffic was analyzed using Wireshark to identify suspicious activity and better understand packet flow.

Analysis & Findings:

  • Real-time capture of network traffic provided valuable data.
  • Several unknown IP addresses were identified communicating with internal servers.
  • Unauthorized HTTP traffic was flagged, suggesting potential data exfiltration.

Challenges & Key Takeaways:

  • The large volume of captured data required effective filtering to identify threats.
  • Utilizing Wireshark’s filters was essential for narrowing down the results.
  • Hands-on experience with packet capture tools is crucial for network security analysis.
Case Study 3: Capturing and Analyzing Network Traffic with Tcpdump

Overview: In this investigation, Tcpdump was used to capture and analyze network traffic, challenging me to master command-line syntax for effective data extraction.

Key Findings:

  • Network packets were successfully captured and filtered to show relevant traffic.
  • Anomalies in network traffic patterns were identified.
  • Initial challenges with command syntax were overcome with practice and troubleshooting.

Lessons Learned:

  • Mastering command-line tools like Tcpdump requires practice and patience.
  • Effective filtering is essential for focusing on security-critical traffic.
  • Command-line tools can complement GUI-based tools like Wireshark in security analysis.
Case Study 4: Investigating a Suspicious File Hash with VirusTotal

Overview: When a financial services company received an alert about a potentially malicious email attachment, VirusTotal was used to analyze the file hash for known malware indicators.

Investigation & Findings:

  • The file hash was analyzed using VirusTotal and compared against multiple malware databases.
  • The file was flagged as a trojan by several antivirus engines.
  • The incident was traced back to a phishing attack targeting employees.

Prevention Strategies:

  • Enhance security awareness training to help employees recognize phishing attempts.
  • Implement robust email filtering solutions to block suspicious attachments.
  • Enforce endpoint security measures to prevent the execution of unverified files.
